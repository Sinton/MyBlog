发布主题前，不要只看首页。主题模板最怕留下原项目的痕迹，或者让使用者 fork 后不知道从哪里开始。

## 内容检查

确认 `src/content` 里只保留通用样稿：

- `docs` 只写主题说明书。
- `articles` 只放展示写作和排版能力的示例专栏。
- `projects` 只放主题样张和组件样张。

如果样稿里出现真实姓名、真实项目、旧站点路线、私有截图或临时迁移说明，就应该改成通用内容。

## 工程检查

确认源码里没有未使用的旧组件、旧数据和旧依赖。可以用这些关键词扫一遍：

```bash
rg -n "临时|私有|旧项目|真实姓名" src README.md package.json
```

如果确实需要保留某个词，确认它是主题说明的一部分，而不是抽离过程留下的痕迹。

## 构建检查

发布前至少运行：

```bash
pnpm check
pnpm build
```

`pnpm check` 用来确认 Astro 和 TypeScript 没有类型问题，`pnpm build` 用来确认所有静态路由都能生成。

## 使用者检查

从一个使用者的角度打开 README 和说明书，确认他能知道三件事：

- 先改哪个配置文件。
- 内容应该放到哪个集合。
- 想改视觉时应该从哪里下手。

如果这三件事讲不清楚，主题看起来再完整也会难用。
